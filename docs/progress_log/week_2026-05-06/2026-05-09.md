### 评估口径厘清：Phase A "area_R 高但 Ch2 recall 低"的算法根因

- 用户复盘 Phase A 失败时直觉提了两个问题："recall 是不是按面积算其实还行？" + "一个大板覆盖多个小板，那些小板应该都算检测成功吧？"
- 第一问的答案：分两层。area_R 0.881（+24pp vs V3-C 0.637）确实涨，但 Ch2 polygon recall@0.3 0.300（−12pp）是跌的——area 视角合格 / count 视角崩盘。bulk_ratio 1.897 是直接证据，area_P 0.465 是同一现象的另一面。
- 第二问的答案：是评估口径与用户直觉不重合。读 `detect_and_evaluate.py:1021` `iou_matching(merge_preds=True)` 实现，确认两堵墙：
  - **墙 1（IoU 阈值）**：1 大 pred 盖 N 个小 GT 时，每个 IoU = 小GT面积 / 大pred面积 → 通常远 < 0.3，根本进不了 candidate 列表。
  - **墙 2（贪心分配）**：line 1097 `available = pidx_set - matched_pred`，第一个 GT 拿走那个大 pred 后被 mark 为 matched_pred，其余 GT 的 available 集合空 → 跳过 → FN。
  - 最好情况 1 TP + (N−1) FN，最坏 0 TP + N FN。Phase A 的 −12pp Ch2 recall 就是这条机制的产物。
- 设计动机：Ch2 polygon recall 度量的是**实例可分性**（能数出装机数量、能做 per-installation 分析），把 N 个板糊成一坨等于丢了这部分信息，metric 应该惩罚。而"大板盖小板算多个 detect"那是 Channel 3 area-aggregate 的视角，Phase A 在 Ch3 里也确实拿到了 area_R 0.881 的功劳——但同时 bulk_ratio 1.9× 也被记过。
- Edge case：如果那 N 个小 GT 其实是同一装机的 sub-array（split_within_gt），那大 mask 概念上对，问题在 GT 该合而没合——正解是 GT-side spatial-merge，不是放宽 matcher（参 `feedback_fragmented_label_semantics.md`）。JHB CBD 25 grid 的 clean_gt 里 SAM_supp + sam_added 都是 sub-array 级（`feedback_sam_tool_ceiling.md`），所以"大 mask 跨 installation 边界"和"大 mask 是对的、GT 该合"两类失败混在一起，没法靠 Ch2 recall 单一维度区分。
- 对清单的意义：今天写的"训练监督分层方案"在 train target 层面收紧 mask 体积（per-source mask weight + 面积自适应 ignore band）+ H 锚点池补 GT-side 干净边界，本身就是从这两个失败方向同时下刀；评估端继续锁 clean_gt（参 `feedback_eval_gt_lock_clean.md`），但下一步可以考虑加一个"被同一个 pred 同时覆盖 ≥2 个 GT"的诊断维度，把 sub-array 类失败和跨 installation 类失败拆开量化。
