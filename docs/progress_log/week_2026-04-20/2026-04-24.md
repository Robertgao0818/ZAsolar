## 2. 路线讨论：高 recall + 分类器 vs V4.3 HN

### 判断

**V3-C（或任何高 recall detector）+ 后处理分类器，比单模型 V4.3 HN 更合理。**

### 论据链

- V3-C 本来就是“高 recall、低 precision”的形态，适合做候选生成器。
- 当前 FP 里很大比例是 solar thermal water heater / roof fixtures / shadow 等语义上可分的 non-PV，小 chip 分类器比 Mask R-CNN detection head 更适合做这类拒识。
- Mask R-CNN 用 hard negatives 教拒识容易稀释训练信号；V4 / V4.1 / V4.2 都说明单靠 HN 很难同时保住 recall 和 precision。
- DeepSolar 的两阶段结构（先分类过滤，再定位/分割）也支持“detector + classifier”的路线。
- V3-C 的 TP / FP confidence 本来已有一定分离，说明后处理层有可利用空间。

### 对 V4.3 HN 的定位调整

V4.3 HN 不再被视为“必须打败 V3-C 的主路线”，而应改成：

- 若 recall 不低于 V3-C、precision 不低于 V4.1，则算成功；
- 否则作为训练数据和 HN 策略的对照实验；
- 主算力优先给 classifier dataset build、backbone ablation、detector-integration evaluation。

### 风险

分类器 non-PV 池目前主要来自 reviewed FP，可能偏向热水器；对架子、道路标线、阴影、skylight 等长尾 FP 覆盖不足。后续 Channel 1 precision audit 应同时作为 classifier failure mining 工具，把新 FP 类型回流进 non-PV 池。

## 5. Hermes / DeepSeek V4 Pro 配置修复

今天还处理了一个非 ZAsolar 主仓、但会影响后续 agent 使用效率的问题：Hermes CLI 会把 `deepseek-v4-pro` 静默归一化成 `deepseek-chat`。

### 问题

用户配置里已经写了：

```yaml
model:
  default: deepseek-v4-pro
  provider: deepseek
  base_url: https://api.deepseek.com
```

但新会话启动时 Hermes 打印：

```text
Normalized model 'deepseek-v4-pro' to 'deepseek-chat' for deepseek.
```

### 排查

确认了以下事实：

- Hermes config 路径：`/home/gaosh/.hermes/config.yaml`
- API key 在：`/home/gaosh/.hermes/.env`
- DeepSeek provider registry 正常，base URL 为 `https://api.deepseek.com/v1`
- 直接调用 DeepSeek `/v1/models` 确认新模型存在：
  - `deepseek-v4-pro`
  - `deepseek-v4-flash`
- 直接调用 chat API 也确认 `deepseek-v4-pro` 可用。

真正 bug 在：

```text
/home/gaosh/.hermes/hermes-agent/hermes_cli/model_normalize.py
```

旧逻辑只把下面两个模型视为 DeepSeek canonical models：

```python
"deepseek-chat"
"deepseek-reasoner"
```

所以 `deepseek-v4-pro` 不在 allowlist 里时会 fall through 到 `deepseek-chat`。

### 修复

本地已把 canonical set 扩展为：

```python
_DEEPSEEK_CANONICAL_MODELS: frozenset[str] = frozenset({
    "deepseek-chat",
    "deepseek-reasoner",
    "deepseek-v4-pro",
    "deepseek-v4-flash",
})
```

验证结果：

```text
deepseek-v4-pro     -> deepseek-v4-pro
deepseek-v4-flash   -> deepseek-v4-flash
deepseek-chat       -> deepseek-chat
deepseek-reasoner   -> deepseek-reasoner
deepseek-r1         -> deepseek-reasoner
deepseek-v3         -> deepseek-chat
```

### 额外确认：reasoning 参数

后续又确认了 DeepSeek V4 Pro 的 reasoning 参数使用 Hermes 通用格式：

```json
{"reasoning": {"enabled": true, "effort": "medium/high/max"}}
```

不是 Kimi 的：

```json
{"thinking": {"type": "enabled"}}
```

并记录了一个长期注意点：Hermes 本地 patch 可能被未来升级覆盖，需要以后升级时检查 `model_normalize.py` 是否仍包含 V4 models。

## 7. Lessons

1. **历史评估问题不能靠记忆答**：今天关于 Li CBD GT 评估连续两次答错，正确做法是先查 `scripts/analysis/`、`results/analysis/` 和对应 progress log。
2. **模型指标必须带口径**：以后写 “recall +7pp / F1 -15pp” 必须同时写 GT 来源、evaluation profile、IoU/cluster 口径、影像源和 model run。
3. **V4-reviewed GT 存在自循环偏差**：用 V4 预测 review 出来的 GT 会天然抬高 V4，独立 Li GT 给出的结论完全不同。
4. **AP50 仍然容易误导**：V4.2 是最新反例；训练阶段必须看 deployment-threshold F1，部署阶段必须看 aggregate metrics。
5. **Agent 环境也要纳入项目效率管理**：Hermes 模型归一化 bug 会让实际模型和预期模型不一致，后续排查“模型怎么变笨了”时要先看启动日志和 provider/model normalization。
