### 训练流程的重要修复

V4.2 的 post-mortem 指出"AP50 单一 metric 误导"是关键失误。今天在 `train.py` 加了 `_compute_f1_at_conf()`：

- 在 COCO eval 完 AP50 之后，**额外算一次 deployment-threshold F1**（conf ≥ 0.85, IoU ≥ 0.5），返回 dict `{ap50, f1@85, p@85, r@85, tp/fp/fn@85}`。
- `best_model.pth` 按 **F1@0.85 选**（不是 AP50），`best_ap50_model.pth` 留作对照。
- Resume 逻辑和 checkpoint 状态里都加了 `best_f1` 字段。
- 这个改动直接回应 V4.2 的 lesson："AP50=0.8371 + 实际部署 F1=0.587"这种 gap 今后训练内就能 catch。

## 5. DeepSolar / DeepSolar++ 重读 → 项目定位重写 (22:45–23:06)

重读两篇论文后，用户用一段话把整个项目的 north star 重新表述：

> **这个项目最终是希望获得一个南非的 DeepSolar++ 数据：有安装时间点、安装位置的 panel data。**

在这个定位下，讨论了三件事：

### 5.1 DeepSolar 的工程套路和我们的区别

- DeepSolar 用 **两阶段 Inception-v3 tile-level 分类 + CAM 弱监督分割**，先砍掉 90% 负样本再看候选。我们是 Mask R-CNN 单阶段 end-to-end，对负样本鲁棒性结构上就弱——这和我们今天定的"后处理二分类器"思路是同一个方向。
- DeepSolar 的 validation 是"每个 census tract 的总装机面积/容量 vs EIA/utility 数据"——**聚合单位内 FP/FN 相互抵消**。这就是 DeepSolar 能在南非同样数据稀缺条件下拿出 R²>0.95 的关键。
- DeepSolar++ 是"高分辨率一次性扫出高置信度安装点 + 低分辨率历史影像回溯安装时间"——这正好对应主仓（aerial）和 bbox 子仓（GEID）的现有分工。

### 5.2 我们有没有类似口径的 external validation

讨论了半天结论是**没有**：
- CT 的 admin data 是点坐标且只集中在市区；Joburg 只有问卷级（household install/no-install），没有落在屋顶上的点。
- 只靠内部检测指标（RA 抽样 + 人工标注）当前阶段只能说明内部一致性，难以说服审稿人。
- 但用户坚持认为应该**彻底转向 DeepSolar++ 的 output shape**：内部指标 + 聚合指标（grid / ward 级的总面积/装机量预测 vs 可得外部对照）。

### 5.3 Ward 单位澄清

用户问"ward 是必须的聚类吗？什么是 ward？" → ward 是南非市政下的行政子单元（council ward），是 StatsSA census / 规划统计的基本单位。结论：**先按 grid 做聚合**，ward 是之后跑实际经济学回归 / 聚类分析时的事。

## 6. Area-aggregate evaluation (DeepSolar-style) 实现 (22:54–00:17)

### 新指标

按 §5 的决策，新增 `scripts/analysis/area_aggregate_eval.py`，对每个 `(region, model_run, grid)` 算：

| Grid 级 | 含义 |
|---|---|
| `pred_total_m²` | 当前 grid 所有预测 polygon 的总面积 |
| `gt_total_m²` | 当前 grid 所有 GT polygon 的总面积 |
| `signed_rel_error` | `(pred - gt) / gt` |

| Run 级 | 含义 |
|---|---|
| `bulk_pred_gt_ratio` | 全部 grid 的 pred 总和 / gt 总和 |
| `bulk_signed_rel_error` | 全局相对误差 |
| `mae_m2_per_grid` | 每 grid 绝对误差平均 |
| `mre_per_grid` | 每 grid 相对误差平均 |
| `frac_grids_within_pm20pct` | 相对误差在 ±20% 以内的 grid 占比 |
| `ols_slope / ols_intercept_m2 / ols_r2` | pred vs gt 的 OLS 回归（新增，用户要求）|

脏数据过滤加了 `_MAX_PLAUSIBLE_POLY_M2 = 20000` 和 `_geometry_finite()`（剔除 NaN/inf bounds）。

### 首轮结果（per_run_summary.csv）

| Region | Model run | n | bulk ratio | MRE/grid | R² |
|---|---|---|---|---|---|
| CT | v3c_targeted_hn_aerial_2025 | 39 | 1.35 | 0.83 | 0.93 |
| JHB | v3c_targeted_hn_aerial_2023 | 50 | 0.93 | 0.34 | 0.92 |
| JHB | v4_aerial_2023 | 50 | 1.07 | 0.35 | 0.89 |
| JHB | v4_2_jhb_ft_aerial_2023 | 10 | **2.19** | **4.79** | **0.44** |

要点：
- V3C 和 V4 在 JHB 50 grid 上聚合误差 < 10%、R² ~0.9，"单检 bounding box 有 FP/FN 但经济学口径下相互抵消" 这个论点 **数据上站得住**。
- V4.2 val-10 的 R²=0.44 + bulk ratio 2.19 确认 §4 的故事：不是 calibration 问题，是 FP 爆炸推高了 pred 总量（过预测 119%）。

## 8. 主仓 / bbox 子仓定位重写 (00:14–00:46)

在 §5 的 north star 下，把两个仓的分工写清楚：

| 仓 | 输入 | 输出 | 目标 |
|---|---|---|---|
| **主仓（ZAsolar）** | aerial（CT 2025, JHB 2023） | 高置信度安装库（installation-level polygon） | 在聚合单位内尽量准确：区域总量对齐是第一位，单检 FP/FN 次要 |
| **bbox 子仓（geid_bbox）** | GEID 历史卫星（多年份） | 同一安装在历史年份的 bbox → 安装时间 | 给主仓的高置信度安装 lib 标注时间戳 |

以这个分工回头看今天的 §2 / §4：
- 二分类器（§2）+ V4.3 HN（§4）目的都是**让主仓输出的"高置信度安装库"在聚合层面和真实装机量对齐**，FP 压不下来聚合数据就会虚高，这直接影响经济学回归的可用性。
- geid_bbox 的 cluster-level bbox 语义（一屋顶多 bbox）正好是"同一安装在时间序列里的像素足迹"，不需要精确到 installation merge。

### 承认的结构性限制

- 南非没有可用的 external validation（admin 点、household survey 都不能直接对齐到屋顶）。
- 只靠 internal RA 抽样 + 标注，确实很难说服审稿人模型的绝对准确性。
- 短期对策：把内部 P/R/F1 **和** 聚合口径（§6 的 bulk ratio / MAE / R²）**一起**报告，让读者看到"单检噪声大，聚合信号稳"的证据。

## 课程修正 / Lessons

1. **AP50 不能作为 training-time 唯一 metric**：V4.2 AP50=0.8371 + 实际 val-10 F1=0.587 的 gap 是硬证据。今天 `train.py` 的 `_compute_f1_at_conf()` 补了这个缺口，best_model 按 F1@0.85 选。
2. **分类器职责不要泛化**：别让一个分类器同时承担"是不是 PV" 和 "是不是热水器"两个任务。保持二分类通用，细分类留给后续专用模块。
3. **Codex 的优化意见值得认真过一遍**：今天的 6 条意见里采纳了 5 条，最关键的一条（registry-driven discovery）直接避免了硬编码路径的维护债。
4. **项目定位要定期重写**：每次读新论文（今天的 DeepSolar++）都可能暴露当前 metric / output shape 和最终目标之间的 gap。不定期做一次 north star 对齐比持续 incremental 优化更重要。
5. **Internal-only validation 的诚实记录**：承认 SA 没有可用 external validation 数据，把聚合指标（R², bulk ratio）当做"内部一致性 + 跨 grid 稳定性"的替代证据，而不是假装它能对齐 ground truth。
6. **Pod 清理要定期化**：500G 限额下 checkpoints / COCO parts / broken mosaics 会很快堆满。建立"每周清一次"的节奏比事到临头手忙脚乱好。
