## 概述

今天把 04-26 确定的 **V3-C-first + classifier cascade** 路线真正落到了实验闭环上。核心结果不是单一模型分数，而是两条互相校正的证据链：

1. **JHB CBD GEID audit pool 暴露出 CT-only classifier 的 OOD 失败**：从 V3-C / V4.2 共同 FP 池中标完 462 个 subtype，其中 119 个其实是 Li GT 漏标 PV。用 CT-trained EffB0 / ConvNeXt / DINOv2 在这 462 个 JHB CBD GEID chips 上 zero-shot 测试，bal_acc 只剩 0.566 / 0.653 / 0.679；在 PV-recall≥0.95 的 cascade-safe 操作点下，non-PV kill 仅 9.6–15.5%，远低于此前 cascade 估算需要的 ~70%。
2. **同一批 v1 classifier 在已有 CT/JHB aerial holdout 上仍有正收益**：用 RunPod 4090 训完 v1 三 backbone 后，在 17-grid holdout（16 grids 有 installation GT）上跑 classifier cascade。threshold=0.5 下三模型都能显著提 precision、净增 F1；ConvNeXt 是 recall budget 下最干净的默认候选，DINOv2 F1/precision 更高但 recall drop 略超 3pp。

因此今天的战略结论要写得更细：**classifier 不是“没用”，而是 CT-only / water-heater-heavy 训练池不能直接迁移到 JHB CBD GEID；v2 必须从 water-heater filter 升级为 multi-subtype FP suppressor，并引入 JHB GEID hard positives + 多子类 hard negatives、按 imagery/domain 分开校准阈值。**

副产物包括：119 个 Li GT supplement polygons（21 grids）、V4.2 subtype propagation、补充 GT 后的 9-cell detector×SAM 矩阵、v2 数据集构建脚本、v2 EffB0 本地训练结果，以及 TP-lost 可视化/标注 gallery。

## 1. Cascade-pool 子类标注闭环

### 1.1 Cascade pool 构建

`scripts/classifier/build_cls_dataset_cascade.py` 从 JHB CBD 25 grids（GEID 2024-02）上的 V3-C / V4.2 SAM mask+box 推理结果出发，对照 Li GT 给每个 polygon 打 `pv/nonpv`，并用 union-find 合并同一 grid 内 V3-C 与 V4.2 的同一物理目标（IoU≥0.3 或 50% containment），记录：

- `source_detector ∈ {v3c, v4_2, both}`
- 初版输出：3913 chips（2054 pv / 1859 nonpv）
- 目标 audit 子集：`detector=v3c AND label=nonpv AND source_detector=both`，共 462 个 V3-C/V4.2 共同 FP 候选

中途修了一个 `chip_path` 对齐 bug：原 manifest 的 chip 路径顺序按 per-grid 写，实际 rows 是 detector-then-grid 顺序，导致图片和 polygon 对不上；已 in-place 修补 manifest.csv / gpkg。

### 1.2 subtype HTML labeler

`scripts/classifier/label_cls_pool_nonpv.py` 做成自含 HTML labeler（base64 嵌图），给 462 个共同 FP 逐个标 subtype。过程中根据实际样本补充了类目，最终 schema 为：

```text

6 hvac | 7 actually_pv_mislabeled | 8 ground_road_marking | 9 corrugated_metal_roof
0 other_unknown
```

可视化质量也同步修正：

- `min_span_m` 从 4m 改 18m，避免小目标被 8× 放大后严重糊掉；
- 红色 bbox 改为红色 polygon outline，保留目标形状；
- 后续 gallery 里又补了一行点击式标注 UI（A/B/C/D + localStorage）。

### 1.3 462 个共同 FP 的 subtype 分布

## 2. GEID mosaic overlap 与跨 grid 去重

标注时发现“审到相同/相似 chip”。检查后确认：JHB CBD 25 grids 使用同一份 GEID mosaic 切 chunked tiles，相邻 grid 之间存在约 35–70m 水平、30–40m 垂直重叠；同一物理目标可能在相邻 grid 各跑一次 detection。

`scripts/classifier/dedup_cls_pool.py` 对 `(detector, label)` 内所有 polygon 做 cross-grid sjoin，IoU≥0.3 或 50% containment 进入同一 union-find group，并添加：

- `dup_group_id`
- `dup_size`
- `is_canonical`

结果：

- 全 pool：3913 rows → 3728 unique physical objects（185 redundant）
- 462 audit core：→ 430 unique

第一版用了 `row.name`，但 sjoin 后 `iterrows()` 返回的是 join index，不是原始 positional index，导致 V3-C non-PV 只识别出 14 groups；改成显式 `i_left / i_right` 后修正。

## 4. V4.2 subtype propagation + 补充 GT 后 9-cell 矩阵

`scripts/classifier/propagate_subtype_to_v42.py` 对 V4.2-side 的 `nonpv-both` polygons 做同 grid IoU≥0.3 配对，把 V3-C 侧 subtype 标签传播过去。

- 441 / 441 V4.2-side records 都找到配对；
- subtype 分布和 V3-C 侧在 ±5% 内；
- 说明 cross-detector pairing 基本是在匹配同一物理目标，可以不重审 V4.2-side。

随后用 Li GT + 119 supplement 重新跑 V3-C / V4.1 / V4.2 × {no SAM, SAM box-only, SAM mask+box} 的 9-cell cluster-level matrix。关键结果：

| 组合 | area F1 | balanced | 备注 |
|---|---:|---:|---|
| **V3-C + SAM box-only** | **0.920** | **0.811** | 仍是 area-F1 / balanced 双冠；04-26 为 0.918 / 0.798 |
| V3-C + SAM mask+box | 0.906 | 0.766 | 比 04-26 的 0.895 / 0.743 略升 |
| V4.2 + SAM box-only | 0.882 | 0.762 | recall 仍高，但 FP 也多 |

| V3-C detector-only | 0.838 | 0.694 | detector-only 明显不如 SAM 后处理 |

结论没有被 supplement GT 反转：**V3-C-first 路线维持，V3-C + SAM box-only 是当前 JHB CBD Li benchmark 上的最强组合。**

## 5. CT-trained classifier 在 JHB CBD GEID audit pool 上的 OOD 测试

把 462 个已人工标 subtype 的 JHB CBD GEID chips 作为 zero-shot OOD test，跑 CT-trained 三 backbone：

| Backbone (v1) | CT val bal_acc | JHB CBD GEID audit bal_acc | non-PV kill @ PV-recall=0.95 |
|---|---:|---:|---:|
| efficientnet_b0 | 0.835 | 0.566 | 9.6% |
| convnext_tiny | 0.869 | 0.653 | 15.5% |
| dinov2_vits14 | 0.891 | 0.679 | 10.2% |

重要解读：

- 不能泛化成“classifier 对 JHB 没用”；这里测的是更难的 **JHB CBD + GEID + Li-missing/FP audit pool**。
- 真正部署时应优先看 recall-constrained non-PV kill，而不是默认 threshold=0.5 或 bal_acc。
- DINOv2 默认阈值/排序能力强，但在 PV-recall≥0.95 的安全约束下不一定比 ConvNeXt 更适合 cascade；ConvNeXt 的错杀分数更可调。
- 70% FP kill 的旧 cascade 估算不能继续用于 GEID CBD；要兑现 GEID/JHB CBD cascade，必须把 JHB CBD subtype-labeled chips 纳入训练池。

## 7. v1 classifier cascade：17-grid holdout 对比

用同一 17-grid holdout 做 threshold=0.5 的 classifier cascade eval：

- CT holdout：7 grids（其中 G1977 本地缺 GT，只能 classification-only）
- JHB holdout：10 grids
- installation aggregate 实际使用 16 grids（6 CT + 10 JHB）

输出：

```text
results/analysis/cls_cascade_holdout/effb0_thr050/
results/analysis/cls_cascade_holdout/backbone_compare_thr050/
  summary.json
  comparison_table.csv
  chip_level_metrics.csv
  summary_metrics.csv
  summary_deltas.csv
  per_grid_metrics.csv
```

16-grid overall 结果：

| model | chip bal acc | ΔP | ΔR | ΔF1 | FP removed | TP lost | filtered P/R/F1 |
|---|---:|---:|---:|---:|---:|---:|---|
| effb0 | 0.8355 | +0.1008 | -0.0365 | +0.0360 | 251 | 61 | 0.8199 / 0.7953 / 0.8074 |
| **convnext** | 0.8691 | +0.1098 | **-0.0287** | +0.0444 | 266 | 48 | 0.8289 / 0.8031 / 0.8158 |
| dinov2 | 0.8914 | **+0.1226** | -0.0335 | **+0.0480** | 292 | 56 | **0.8416** / 0.7983 / **0.8194** |

解读：

- ConvNeXt 是当前最干净的 default candidate：F1 增益明显，recall drop 2.87pp，刚好守住 3pp 预算；
- DINOv2 precision / F1 最高，但 recall drop 3.35pp，略超默认预算；
- EffB0 可作为轻量 baseline，但不再是最强候选；
- 当前 v1 classifier 并非“实质失败”：在 16-grid holdout 上大约用 48 个 TP loss 换回 266 个 FP removal（ConvNeXt，约 5.5:1），净 F1 是正的。

区域差异：CT precision gain 大但 recall drop 也更大，可能与 CT hard negative / water heater 更密集有关；JHB aerial holdout 的 recall drop 小于 CT。这个结果支持“按 imagery/domain 分开校准阈值”，而不是共用 0.5。

## 8. TP-lost gallery 与错杀审计

为了判断 recall drop 是“合理 trade-off”还是“明显 PV 被模型错杀”，新建：

```text
scripts/analysis/build_tp_lost_gallery.py
results/analysis/cls_cascade_holdout/tp_lost_gallery/convnext/index.html
results/analysis/cls_cascade_holdout/tp_lost_gallery/dinov2/index.html
```

功能：

- 为 ConvNeXt / DINOv2 被 classifier 砍掉但和 GT 匹配的 TP-lost 生成 chip gallery；
- HTML 中加入 A/B/C/D 一行点击式标注 UI，localStorage 保存；
- 类别：A 灰区、B 明显 PV（错杀）、C GT 错、D 检测错。

用户快速标注结果：

| 类别 | ConvNeXt | DINOv2 |
|---|---:|---:|
| 总计 | 51 | 56 |
| **B 明显 PV（错杀）** | **43 (84%)** | **42 (75%)** |
| A 灰区 | 2 | 4 |
| C GT 错 | 5 | 2 |
| D 检测错 | 0 | 1 |
| 未标 | 1 | 7 |

关键发现：

- gallery 中大部分 TP-lost 肉眼看确实是 PV，说明 v1 recall drop 不是完全合理的 “heater/PV entanglement 代价”，而是真有一批 PV 外观没有被训练集覆盖；
- 但因为 FP/TP 兑换比仍然很好（ConvNeXt 约 5.5:1），v1 cascade 仍可作为短期有效 baseline；
- ConvNeXt 与 DINOv2 失败模式不完全重叠，后续可测试 `max(score_convnext, score_dinov2)` 或 “kill only if both models below threshold” 的 high-recall ensemble，但必须以 recall-constrained FP kill 曲线验证，不能直接凭错杀交集上线。
