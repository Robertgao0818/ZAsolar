## 主仓进展：JHB CBD / Vexcel 25-grid review 完成

本周主仓最核心的工作是完成 JHB CBD 的 Vexcel 样本 review。具体来说，不只是“看完 25 个 grid”，而是把这批样本整理成后续能复用的数据资产：

- 对 25 个 grid 的模型预测进行人工核查，区分明显正确、明显错误和需要补充的区域；
- 对漏检位置进行补标，尤其关注模型没有完整覆盖的屋顶太阳能区域；
- 清理 GT 中不一致或容易误读的部分，避免把 review 产物、补标碎片和真实安装数量混在一起；
- 核对不同来源的标注和预测文件，确保后续训练 / 验证能追溯来源；
- 明确这批数据会作为 JHB / Vexcel fine-tune 的关键输入。

对经济学同事来说，这一步的意义是：我们不是只在提高模型分数，而是在降低空间经济分析中的测量误差。后续如果要比较不同社区、不同建筑类型或商业区与住宅区的 rooftop PV 普及程度，JHB CBD 这批 clean GT 是判断模型是否系统性偏差的重要依据。

## GT 清理完成：从 review 结果到可训练数据

本周另一个重要产出是 **GT 清理完成**。这里的 GT 不是单纯一份人工画图文件，而是经过多轮核对后的训练 / 验证依据。

清理工作包括：

- 合并人工 review 中确认正确的预测；
- 补充模型漏掉但人工确认存在的 PV；
- 剔除明显不是 PV 的误报；
- 对大型工商业阵列做额外检查，避免只保留模型抓到的一小块；
- 保留来源信息，方便之后解释某个样本来自原始标注、人工 review 还是补标。

这部分工作量比较大，但对后续结果可信度很关键。如果 GT 本身混乱，fine-tune 可能学到错误信号，评估也可能把真实 PV 错当成模型误报，或者把漏检问题掩盖掉。

## 本周主要发现：JHB CBD 的误差不是均匀的

本周 review 和 GT 清理后，误差结构比之前更清楚：

- 住宅小型 rooftop PV 的迁移效果相对可接受；
- CBD / 工商业屋顶的大面积阵列仍是主要风险；
- 模型有时能找到大阵列的一部分，但没有覆盖完整面积；
- 如果只看“有没有找到一个 polygon”，会低估这个问题；
- 对经济学用途来说，面积漏检比单个边界误差更严重，因为它会直接影响 capacity proxy。

因此，下一轮 fine-tune 不能只随机增加更多住宅样本，而要重点补 JHB / Vexcel 场景，特别是工商业大阵列。这也是本周完成 clean GT 的直接价值：它让后续 fine-tune 有了更明确的训练目标。

## 评价口径：继续面向 aggregate inventory

本周继续沿用 V1.4 的方向：把主目标放在 **aggregate inventory quality**，即每个 grid 内 rooftop PV 总量是否可信。

经济学研究通常关心的是：

- 某个 grid / suburb / ward 的 PV 普及程度；
- 不同收入、建筑形态或用地类型区域之间的差异；
- 商业区和住宅区的装机面积比较；
- 未来和电价、停电、收入、住房等变量合并后的空间相关性。

这些问题不要求每个单独 polygon 都完美，但要求聚合后的面积和数量不能系统性高估或低估。因此，本周的 review、GT 清理和大阵列诊断，本质上都是为了让后续 grid-level 数据更可信。

## 子仓进展：`solar_backdating` 初步落地可用

子仓 `solar_backdating` 本周完成了初步拆分和可运行闭环。它和主仓的分工更清楚了：

- 主仓 `ZAsolar`：回答“现在某个地方有多少 rooftop PV”；
- 子仓 `solar_backdating`：回答“这些 rooftop PV 大约是什么时候装上的”。

本周子仓的主要进展是把安装时间回溯从主仓中拆出来，并把问题定义收窄到更稳健的 **anchor-presence** 任务。也就是说，不是在每一张历史影像上重新检测所有太阳能板，而是先由主仓提供一个当前已经确认存在的 PV 位置，然后在历史影像中判断这个位置附近在某个时间点是否已经能看到 PV。

这个设计对早期阶段更合适，因为历史影像通常存在分辨率、角度、阴影和配准误差。如果直接做历史图上的完整检测，样本量和误差都会比较难控制；而 anchor-presence 更接近经济学需要的结果：安装大约发生在哪个时间段。

本周子仓已经能跑通基本流程：

- 从主仓输出的 PV anchor 出发；
- 整理对应位置的历史影像；
- 对不同历史时间点判断 PV 是否已经出现；
- 将 `present` / `absent` / `ambiguous` 结果整理成安装时间区间；
- 输出可供人工复核的 QA 文件和中间结果；
- 保留 confidence / ambiguous 标记，避免把不确定影像强行转成精确日期。

本周做过的小样本验证包括：

- 使用少量 anchor 跑通 historical imagery → visual review → install interval 的闭环；
- 检查 Gemini / MLLM 对历史影像的判断是否稳定；
- 对容易误判的样本加入更清楚的视觉提示，例如标出 anchor 附近需要检查的区域；
- 对 ambiguous case 保留人工复核入口，而不是直接丢弃或强行分类；
- 初步确认 pipeline 可以作为后续人工标注和扩大样本的基础。

目前这条线仍是前期阶段，重点不是马上生产大规模时间面板，而是验证流程是否可用、标注口径是否稳定、历史影像是否足以支持 install-date inference。从本周情况看，pipeline 初步可用，但还需要更多 anchor、更多人工复核和更清楚的 ambiguous case 处理。

## 子仓为什么重要

如果只有主仓的静态地图，我们能知道某个地区现在有多少 rooftop PV；但很多经济学问题需要时间维度，例如：

- 能源危机或停电频率变化后，PV adoption 是否加速；
- 高收入社区是否更早安装；
- 商业区和住宅区的采用时间是否不同；
- rooftop PV 是否沿着空间邻近关系扩散。

因此，子仓的目标不是替代主仓检测，而是在主仓 inventory 之上增加 adoption timing。当前的小样本验证是为了确认这条时间回溯路线是否值得扩大。

## 本周交付脉络

本周的推进可以理解为从“样本核查”到“下一轮训练输入”的完整准备过程，而不是单独完成某一个脚本或指标。主仓侧围绕 JHB CBD / Vexcel 数据形成了较完整的数据闭环：

- 先完成 25-grid 人工 review，确认哪些预测可以信、哪些是误报、哪些地方需要补标；
- 再把 review、补标和既有标注整理成 clean GT，让后续 fine-tune 和评估有一致依据；
- 同时根据 review 结果识别主要误差来源，确认 CBD / 工商业大阵列是下一步最需要改善的部分；
- 最后把这批数据转化为 2026-05-06 fine-tune 的输入和评估基准。

子仓侧则完成了从 repo 拆分、任务定义到小样本闭环验证的早期搭建。整体上，本周的进展把两个方向都往前推了一步：主仓更接近可用于经济分析的空间库存数据，子仓开始验证能否在这个库存之上增加安装时间维度。
