# 工作记录 2026-04-12 (Sat)

**工作时间**: 约 6 小时

## Cross-review harness 重构 + Pipeline 数据层重新设计

### 概述

今天的工作分两大块：一是把 agent 协作框架（harness）和标注注册系统升级到多城市架构，二是按 pipeline redesign V1.2 方案落地数据导入和数据标注模块。核心目标是让整个系统能平滑地从"只做开普敦"扩展到"开普敦 + 约堡（以及未来更多城市）"，而不需要到处改硬编码的路径和参数。

---

### 1. Harness 与标注注册系统重构

**动机**：此前的 harness（agent 做代码审查和实验审查时参照的检查清单）和标注系统都是围绕开普敦单城市设计的。约堡数据已经进入流程后，审查规则、标注路径、坐标系等都需要适配多城市。

**完成内容**：

- **context-map 更新**：新 agent 接手时读的"地图"文件全面更新——标注目录从 `cleaned/` 改为 `Capetown/` 和 `Joburg/` 两层结构；新增 `region_registry.py` 和 `annotation_loader.py` 两个核心模块的说明；添加 Two-Axis Model 速查表和"添加新城市"步骤清单
- **code-review checklist 重构**：原来的检查项"是否把 Cape Town 假设写死到 JHB"替换为更精确的规则——路径必须通过统一函数获取、坐标系必须查询不能硬编码、标注升级必须符合 Two-Axis Model
- **experiment-review checklist 更新**：新增约堡 CBD 25 格测试集（`jhb_cbd_reviewed_25`）作为 secondary benchmark；要求跨城市实验必须分城市报告，不允许把开普敦和约堡指标混在一起当结论

**为什么重要**：harness 是质量控制的自动化层——当 AI agent 帮忙做代码审查或实验分析时，它会按照这些清单逐项检查。如果清单还是单城市思路，agent 就会漏掉多城市相关的问题。

### 2. Region Registry 和 Annotation Loader（新建）

**动机**：之前每个脚本（推理、导出 COCO 数据集、标注审查等）都自己写一套"去哪里找 tile 文件、去哪里找标注文件"的逻辑。城市多了之后这种做法不可维护。

**完成内容**：

- **region_registry.py**（新建）：读取 `regions.yaml` 配置文件，提供统一的查询接口——给一个格子编号，返回它属于哪个城市、用什么坐标系、tile 放在哪、标注放在哪。所有脚本通过这一个入口获取路径信息
- **annotation_loader.py**（新建）：统一的标注发现和加载模块——扫描所有注册城市的标注目录，返回标准化的标注数据。解决了开普敦标注（SAM2 格式）和约堡标注（V4-reviewed 格式）文件命名和图层结构不同的问题。设计了过渡机制：registry 中已注册的格子优先，未注册的通过目录扫描兼容
- **regions.yaml 扩充**：约堡从原来的 6 行占位信息扩充为完整的 31 格注册（6 个 legacy + 25 个 CBD batch1），包含标注文件路径、标注数量、备注；两个城市都加了 `paths` 块（tile/results/annotations/task_grid 的标准路径）和 `grid_id_pattern`（格子编号的正则表达式）

### 3. 核心脚本多城市适配

**动机**：有了 registry 之后，现有脚本需要改造为"通过 registry 获取路径"而非"自己拼路径"。

**完成内容**：

- **grid_utils.py 重构**：所有路径解析函数（找 tile、找标注、找输出目录）增加 `region=` 参数，内部委托给 `region_registry`。保留了兼容回退逻辑，确保旧脚本不加参数也能正常工作
- **detect_and_evaluate.py**：推理主入口新增 `--region` 命令行参数，传递给 grid context 设置。约堡格子现在可以直接 `--grid-id G0922 --region jhb` 运行
- **export_coco_dataset.py**：COCO 数据集导出改为通过 `annotation_loader` 发现标注，新增 `--regions` 参数支持指定导出哪些城市的数据。核心导出逻辑重构为 `build_base_coco()` 函数，方便未来被 pipeline builder 程序化调用
- **review_detections.py**（标注审查 GUI）：全面多城市适配——tile 路径通过 `resolve_tiles_dir()` 解析；新增 tile 元数据缓存避免重复读取；改进了预测和 tile 边界的交叉匹配逻辑，解决了约堡 tile 地理范围与格式坐标不完全对齐的问题
- **sam_fn_review.py**（SAM 漏检复查 GUI）：坐标系从硬编码改为按城市查询；tile 路径支持开普敦和约堡两个目录；SAM 推理增加 tile 缓存（避免重复读取同一张 tile）和 embedding 缓存（避免对同一张 tile 重复计算 SAM 特征），GUI 响应速度显著提升

### 4. 训练集谱系追踪体系

**动机**：当前有多个训练集和多个模型版本，但"某个模型是用哪个数据集、什么参数训练的"只存在于对话记录和零散的笔记中。需要一个结构化的记录。

**完成内容**：

- **training_sets.yaml**（新建）：记录每个训练集的"配方"——用了哪些城市的数据、包含哪些类型的标注、负样本比例、hard negative 策略、输出路径。目前记录了 v3c_base、v3c_targeted_hn、v4_base、v4_1_hn 四个训练集
- **model_registry.yaml 增强**：每个模型条目新增 `training_set_id`（指向 training_sets.yaml）、`region_scope`（覆盖哪些城市）、`postproc_config`（后处理配置）。实现了从"模型 → 训练集 → 标注来源"的完整追溯链

### 5. Benchmark 配置扩充

- `post_train.yaml` 新增 `jhb_cbd_reviewed_25` 测试集（25 个约堡 CBD 格子，role=secondary），用于快速验证模型在新城市的迁移效果
- 原有 `jhb_transfer_6` 补充了 `region: jhb` 标签和 vintage gap 风险说明

### 6. 标注规范（ANNOTATION_SPEC.md）正式纳入 Two-Axis Model

将此前在对话中讨论确定的 Two-Axis 标注质量模型正式写入规范文档。关键补充：

- A 轴（语义符合度 A1/A2/A3）和 B 轴（来源 H/R/S/G）的完整定义表
- "SAM 作为工具 vs SAM 衍生标注"的关键区分——同样用了 SAM，人主动发起的标注（H）和模型预测经 SAM 重切的标注（S）在数据谱系上有本质区别
- 新标注来源入库流程（onboarding procedure）：分类 → 默认赋值 → 文件规范 → 注册 → 验证
- 质量等级（T1/T2）与 Two-Axis Model 的关系：T1 仅限 A1 级标注，不允许仅凭标注来源自动升级
